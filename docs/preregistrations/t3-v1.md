## 1. Primary endpoint and primary hypothesis

- **Primary endpoint (T3):** the continuous **test-pass fraction**
  `score = passed_tests / total_tests ∈ [0, 1]`, computed by the frozen
  R1-4 scorer **executed in the sandbox** (`score_t3_sandboxed`; spec
  `docs/t3-scoring-spec.md`). Parse/execution failures do **not** become a
  mid-scale value: they are flagged (`parse_failure` / `execution_failure`)
  with score 0.0 and distinct `failure_reason` codes.
- **Primary hypothesis:** **H1 on T3 — C4 > C1⁺** (structure at constant
  information, transformation task).
- **Aggregation:** per-function mean across the two frozen models, then paired
  differences across functions (identical to T1/T2).
- **Per-model analyses are exploratory only.**

## 2. Hypothesis family and correction

Same **pre-existing 9-test family** as T1/T2 (Constitution v0.3 / governance);
T3 supplies the three remaining cells (H1/T3, H2/T3, H3/T3).

- **Family size: m = 9.** Holm–Bonferroni step-down at α = 0.05, via the frozen
  `analyze_confirmatory.py`.
- Test: one-sided Wilcoxon signed-rank (`alternative='greater'`,
  `zero_method='wilcox'`). Effect size: rank-biserial `r_rb` with 95% percentile
  bootstrap CI (seed 20260427, 10 000 resamples; R2-1).
- **Family-completion note:** R5-4 populates the final three cells, so the
  Holm decision computed at R5-4 is the **first complete** family-wise
  correction over all nine cells (T1 was added at R5-2). All raw per-test
  quantities are reported regardless.

## 3. Dataset, ground truth, power, and the ceiling risk

### 3.1 Dataset and pre-declared exclusions

- The **26** confirmatory (`dataset_tier = "full"`) functions that have a
  **frozen post-transform test suite** (`tasks.T3.available = true`,
  `data/ground_truth/tests/{func_id}_t3_test.py`).
- **Pre-declared exclusions (n: 30 → 26):** `peak_valley_indices`,
  `segments_above_threshold`, `stable_partition`, `group_by_key` have **no**
  T3 transform spec and **no** frozen post-transform suite. They are excluded
  for a purely structural reason (no ground truth exists), decided **before**
  any T3 model call and independent of any outcome. Authoring + freezing four
  new suites is deferred (would be a future re-registration); the R1-4
  enrichment contingency (functions with < 4 tests) does **not** trigger — every
  retained suite has ≥ 8 tests.
- Pilot anchors (`clamp`, `count_vowels`, `is_sorted`) remain segregated.

### 3.2 Realised anti-ceiling resolution (R1-4 §4 on the 26 retained suites)

- Test counts per suite: min 8, max 13, mean 10.69, median 11; distribution
  {8: 2, 9: 4, 10: 6, 11: 4, 12: 8, 13: 2}; none < 4.
- Attainable values per function = total + 1 ∈ [9, 14]; **pooled 53 distinct
  attainable score values** in [0, 1]. The *attainable* resolution cannot
  reproduce the T2 0–3 saturation.

### 3.3 Ceiling **realisation** risk (pre-declared, T3-specific)

Attainable resolution ≠ realised resolution. The R1-5 held-out pilot found the
sandboxed T3 scorer **saturated at 1.0** on all three (easy) pilot anchors for
both models. If the frozen models also pass all tests on the 26 (harder)
confirmatory transforms, T3 scores will concentrate near 1.0, the paired
differences will be mostly ties, and the **effective Wilcoxon n** will collapse
— the same mechanism that defeated H1/T2 (paper §5.3), but at the top of the
scale. This is **pre-declared as a primary threat to T3 power**. It is reported,
not corrected away: at R5-4 we report the realised score distribution, the
per-condition tie rate, `n_+ / n_- / n_=`, and the effective n alongside the
test, exactly as for T2. A non-rejection under saturation is interpreted as a
measurement-ceiling outcome, not as evidence for H0.

### 3.4 Power analysis (Constitution v0.3 decision 5)

Same method as T1 (`scripts/r5_power_analysis.py`), at the frozen **n = 26**.
Target power 0.80.

- **Smallest detectable effect at n = 26:** dz ≈ **0.68** at the corrected
  threshold (α′ = 0.05/9), dz ≈ **0.50** at nominal α = 0.05 (both power 0.80).
- Minimum n by effect size (corrected, power 0.80): dz = 0.5 → n ≈ 48,
  dz = 0.6 → n ≈ 34, dz = 0.8 → n ≈ 19.
- **Justification of n = 26 (pre-declared).** The 26-function set is fixed by the
  frozen ground truth. It is powered ≥ 0.80 only for **large** paired effects
  (dz ≳ 0.68 corrected); it is **pre-declared underpowered for small-to-medium
  effects**, and additionally subject to the §3.3 ceiling risk that can suppress
  the *realised* effect regardless of nominal power. n is not expanded because
  the suites are frozen and authoring four more is deferred. A non-rejection is
  read against this power profile and the ceiling diagnostics, never as
  evidence of absence.

## 4. Transformation task, models, and execution safety

### 4.1 Transformation task definition (exact)

- For each function the instruction is the frozen `instruction` field of its
  `t3-transform-v1` spec (`data/ground_truth/transforms/{func_id}.json`), loaded
  verbatim into the frozen T3 prompt template (`runner._T3_TEMPLATE`,
  `PROMPT_VERSION = "pilot-v1"`).
- **Demanded output:** the complete transformed Python function, in a Python
  code block (the template asks for "the full function definition"). The scorer
  extracts the fenced block / `def {func_id}` and runs the post-transform suite.
- Representation: for T3 the runner uses the **correct** (non-buggy)
  representation of each function in each of the five conditions.
- Prompt wording is identical across conditions; only the representation block
  varies.

### 4.2 Models and sampling

Frozen pair: `gpt-5.4` (OpenAI) and `claude-opus-4-6` (Anthropic),
`temperature = 0.0`, held fixed across all five conditions (v0.3 decision 4;
neither is in the temperature-unsupported set). `max_tokens = 2048`. Crossing:
5 conditions × 2 models × 26 functions = **260 model calls**. API snapshot dates
recorded in the R5-4 run artifacts.

### 4.3 Execution safety (pre-registered)

- Model-generated code is executed **only** inside the R1-4 sandbox
  (`score_t3_sandboxed` → `run_t3_tests_sandboxed`): a child subprocess with a
  **10 s wall-clock timeout** (`T3_SANDBOX_TIMEOUT_S`). On expiry the child is
  killed and the cell is recorded as `execution_failure` /
  `failure_reason.code = execution_timeout`.
- The collection driver `scripts/run_confirmatory.py --task T3` rebinds the
  in-runner T3 scorer to `score_t3_sandboxed` (the frozen `runner.py` is **not**
  edited); verified end-to-end (correct transform → 1.0; infinite loop →
  `execution_timeout`).
- Resource caps beyond wall-clock isolation (`setrlimit`) are an optional R1-4
  hardening and are **not** required here; the wall-clock fence is the
  pre-registered guarantee.

## 5. Exclusion rules and pre-declared sensitivity analyses

### 5.1 Exclusion / failure handling

- `parse_failure` (no code block / syntax error / wrong-or-missing target
  function) and `execution_failure` (module-load / timeout) score 0.0 with
  distinct codes and are **retained** as honest 0.0 in the primary analysis (a
  transform that does not run is a real failure of the task), **but** are also
  tracked separately so the MNAR sensitivity (§5.2.1) can re-test excluding
  them. `scoring_error` / `zero_total_tests` cells (harness faults, not model
  faults) are treated as missing, not as 0.0.

### 5.2 Sensitivity analyses (pre-declared, run at R5-4)

1. **Parse/execution-failure MNAR check** (R2-2 style, required by the issue):
   report `parse_failure` + `execution_failure` counts per condition × model;
   re-test H1/T3 (a) with failures retained as 0.0 (primary) and (b) excluding
   failed cells (complete-case). Divergence between (a) and (b) is reported; a
   conclusion that flips is flagged. This guards the L3-05 missing-not-at-random
   risk specifically for code that fails to run.
2. **Ceiling / tie diagnostics** (§3.3): realised score distribution, distinct
   realised values, per-condition tie rate, `n_± / n_=`, and effective Wilcoxon
   n, reported alongside the test.
3. **Per-model breakdown** (exploratory): H1/H2/H3 on T3 within each model,
   uncorrected (model × representation interaction, E5).
